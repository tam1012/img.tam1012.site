## 5. Kiểm tra mã nguồn

### 5.1 Cấu trúc

Dự án sạch, tổ chức hợp lý. Code theo đúng mô tả trong AGENTS.md.

### 5.2 Các vấn đề phát hiện

#### ~~🔴 Quota khách~~ → ĐÃ XÁC NHẬN: Code 50 là đúng, đã cập nhật docs cho khớp.

#### ~~🟡 nginx timeout~~ → ĐÃ SỬA: Tăng `proxy_read_timeout` từ 120s lên 300s trên VPS.

#### 🟡 NEW: Gemini trả lỗi không rõ ràng khi bị chặn nội dung gợi cảm

**Hiện tượng:** Edit ảnh gợi cảm bằng Gemini → lỗi "Provider không trả về ảnh qua chat completions" — không nói lý do thật.  
**Nguyên nhân:** Gemini trả HTTP 200 nhưng chỉ có text từ chối, không có ảnh. Code cũ không đọc text đó.  
**Đã sửa:** `extractChatImage` giờ trích xuất text Gemini trả về để hiển thị lý do từ chối thật.

**Đề xuất:** Tăng `proxy_read_timeout` lên `300s` (5 phút).

#### 🟡 MEDIUM: `client_max_body_size` thấp hơn tài liệu

**Tài liệu:** AGENTS.md ghi `client_max_body_size 50M`  
**Thực tế:** nginx config chỉ có `client_max_body_size 20M`

20MB đủ cho hầu hết ảnh đơn lẻ, nhưng nếu upload nhiều ảnh PNG lớn cùng lúc có thể vượt quá. Đề xuất tăng lên 50M như đã định.

#### 🟡 LOW: API route bị redirect thay vì trả 401 JSON

Middleware (`src/middleware.ts`) redirect tất cả request không có session về `/login`, kể cả API routes. API consumer (bot, script) sẽ nhận HTTP 307 thay vì JSON error `{ error: "Chưa đăng nhập" }`. 

Trong thực tế, API routes vẫn có `requireAuth()` check riêng và trả 401 JSON nếu session tồn tại nhưng không hợp lệ. Nhưng nếu hoàn toàn không có cookie, API trả redirect 307 gây khó cho script.

**Đề xuất:** Middleware nên trả 401 JSON cho request bắt đầu bằng `/api/` thay vì redirect.

#### 🟢 LOW: `img-role` cookie không httpOnly

Middleware dùng `img-role` cookie để chặn guest vào `/settings` mà không cần decrypt session. Điều này lộ role ra JavaScript. Tuy nhiên tác động thấp vì:
- API routes vẫn kiểm tra session thật (iron-session) khi cần quyền admin
- Middleware redirect trang settings chỉ là UI convenience

#### 🟢 LOW: Tài liệu kiến trúc lỗi thời

`docs/architecture.md` mô tả provider abstraction bằng interface và các file riêng (`providers/openai.ts`, `providers/google.ts`), nhưng thực tế đã gộp thành 1 file `providers/custom.ts`. Cần cập nhật tài liệu.

## 10. Kết luận

**Tổng quan:** Dự án IMG Studio hoạt động tốt, triển khai sạch, bảo mật cơ bản vững. Code gọn gàng, dễ bảo trì.

**Điểm mạnh:**
- Cả 4 model đều tạo ảnh OK, 3/4 model chỉnh sửa OK
- Auth 2 role hoạt động đúng
- API key được mask khi trả về client
- Dark theme UI nhất quán, tiếng Việt đầy đủ
- Guest bị chặn đúng khỏi Settings
- Deploy CI/CD tự động, volume persist data

**Điểm yếu còn lại:**
- Thiếu rate limiting và backup tự động
- Một số tài liệu kiến trúc lỗi thời so với code

**Đã sửa trong phiên audit:**
- Docs quota sửa khớp code (50 ảnh/ngày)
- nginx timeout tăng 120s → 300s
- Gemini edit lỗi mơ hồ → hiển thị lý do từ chối thật

**Điểm tổng thể: 8.5/10** — Dự án cá nhân chất lượng tốt, vận hành ổn định. Các vấn đề chính đã được sửa trong phiên này.
